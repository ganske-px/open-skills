# Decision-Making Anti-Patterns

> Common mistakes that undermine effective decision-making and how to avoid them.

## Cross-Framework Anti-Patterns

### 1. Framework Worship
**Pattern:** Applying frameworks rigidly without judgment.

**Symptoms:**
- Using full cycle for simple decisions
- Forcing every decision through all 5 frameworks
- Valuing process over outcome

**Recovery:**
- Match framework weight to decision stakes
- Skip phases that don't add value
- Focus on outcome quality, not process compliance

---

### 2. Analysis Paralysis
**Pattern:** Endlessly gathering information without deciding.

**Symptoms:**
- Repeated requests for "more data"
- Matrix criteria keep expanding
- Pre-mortem never ends
- Decision deadline keeps slipping

**Recovery:**
- Set explicit decision deadline
- Define "good enough" information threshold
- Timebox each framework phase
- Ask: "What would change our mind?"

---

### 3. Process Theater
**Pattern:** Going through motions without genuine engagement.

**Symptoms:**
- Frameworks completed but not discussed
- Decision already made, process is justification
- Stakeholders not actually consulted
- Documentation exists but wasn't used

**Recovery:**
- If decision is made, acknowledge and skip to execution
- Ensure right people are genuinely involved
- Value the thinking, not the artifact

---

### 4. Framework Avoidance
**Pattern:** Avoiding structure for "intuition" on complex decisions.

**Symptoms:**
- High-stakes decisions made casually
- "We've always done it this way"
- Gut feel without supporting analysis
- No documentation of reasoning

**Recovery:**
- Recognize when stakes warrant structure
- Use quick decision workflow as minimum
- Document rationale even for intuitive choices

---

## Cynefin Anti-Patterns

### 5. Treating Complex as Complicated
**Pattern:** Believing analysis will reveal "the answer" when experimentation is needed.

**Symptoms:**
- Extended analysis without progress
- Expert disagreement treated as solvable by more data
- Surprise when "proven" approaches fail
- Frustration that cause-effect isn't clear

**Recovery:**
- Recognize when you're in Complex domain
- Design safe-to-fail probes
- Accept that patterns emerge through action
- Value learning over planning

---

### 6. Treating Complicated as Clear
**Pattern:** Applying simple rules to nuanced situations.

**Symptoms:**
- "Just follow the process"
- Ignoring expert input
- Edge cases dismissed as exceptions
- One-size-fits-all solutions

**Recovery:**
- Consult domain experts
- Accept multiple valid approaches exist
- Document context for why this case is different

---

### 7. Staying in Disorder
**Pattern:** Failing to classify, remaining stuck.

**Symptoms:**
- "It depends" without committing
- Endless debate about problem nature
- Paralysis from uncertainty
- No action while waiting for clarity

**Recovery:**
- Force a classification (even if wrong)
- Default to Complex if uncertain
- Start with probe to learn more
- Revisit classification after action

---

### 8. Complacent Drift
**Pattern:** Treating evolving Complex situations as Clear.

**Symptoms:**
- "We've always done it this way"
- Ignoring signals of change
- Metrics unchanged despite shifting context
- Surprise crises from "stable" situations

**Recovery:**
- Regularly question domain classification
- Monitor for weak signals
- Build in periodic reassessment
- Challenge assumptions explicitly

---

## RAPID Anti-Patterns

### 9. Multiple D's
**Pattern:** Decision by committee with no single accountable person.

**Symptoms:**
- "The leadership team will decide"
- Endless meetings without resolution
- Everyone has to agree
- Decisions unmade or relitigated

**Recovery:**
- Insist on single D
- Escalate if D unclear
- D can consult but must own decision

---

### 10. Everyone is A
**Pattern:** Too many people have veto power.

**Symptoms:**
- Any objection blocks progress
- Change requires universal agreement
- Speed sacrificed for consensus
- Fear of offending anyone

**Recovery:**
- Reserve A for genuine compliance needs
- Distinguish A (veto) from I (input)
- Accept that good decisions may have objectors

---

### 11. Silent P
**Pattern:** Performers not involved in decision, then resist.

**Symptoms:**
- Implementation resistance
- "That won't work" after decision made
- P relitigates instead of executes
- Execution quality suffers

**Recovery:**
- Include key P voices as I
- Communicate decision rationale to P
- Provide escalation path for genuine blockers
- Separate disagreement from new information

---

### 12. I Overload
**Pattern:** Consulting so many people that nothing moves.

**Symptoms:**
- Stakeholder list keeps growing
- Input contradicts, causing confusion
- Analysis paralysis from too many opinions
- D overwhelmed by volume

**Recovery:**
- Limit I to essential perspectives
- Weight input, don't just collect
- D must filter, not incorporate all

---

## Decision Matrix Anti-Patterns

### 13. Gaming the Matrix
**Pattern:** Manipulating criteria or weights to justify preferred option.

**Symptoms:**
- Criteria added that favor one option
- Weights adjusted after scoring
- "Surprising" result that confirms bias
- Matrix built to prove, not discover

**Recovery:**
- Define criteria before evaluating
- Lock weights before scoring
- Have different people own criteria vs scoring
- If result doesn't feel right, examine biases

---

### 14. False Precision
**Pattern:** Treating small score differences as meaningful.

**Symptoms:**
- 3.72 vs 3.68 treated as clear winner
- Decimal points carried through
- No acknowledgment of uncertainty
- "The numbers say" as final argument

**Recovery:**
- Use whole numbers
- Acknowledge close calls as close
- Sensitivity analysis on tight races
- D can override with rationale

---

### 15. Criteria Creep
**Pattern:** Too many criteria dilute important factors.

**Symptoms:**
- 15+ criteria in matrix
- Every stakeholder adds their criterion
- Weights so small nothing matters
- Analysis takes days

**Recovery:**
- Maximum 8 criteria
- Merge related criteria
- Minimum 10% weight to include
- Ask: "Would this change the outcome?"

## Pre-Mortem Anti-Patterns

### 16. Superficial Pre-Mortem
**Pattern:** Only identifying obvious, surface-level risks.

**Symptoms:**
- 3-4 generic risks listed
- No uncomfortable possibilities raised
- Everyone agrees too quickly
- No one mentions the elephant

**Recovery:**
- Require 10+ failure reasons
- Force uncomfortable scenarios
- Anonymous submission of risks
- Ask: "What would enemies say?"

---

### 17. Pre-Mortem as Veto
**Pattern:** Using risk identification to block unwanted decisions.

**Symptoms:**
- Every option has "fatal" flaws
- L×I scores inflated
- No constructive mitigations offered
- Pre-mortem always recommends "no-go"

**Recovery:**
- Balance risk identification with mitigation
- Challenge inflated scores
- Require mitigation proposals for each risk
- Separate "I don't like it" from genuine risk

---

### 18. Ignoring Pre-Mortem
**Pattern:** Conducting pre-mortem but not acting on findings.

**Symptoms:**
- High risks documented but no mitigations
- Same risks appear in post-mortem
- "We knew this might happen"
- Documentation without action

**Recovery:**
- Require mitigations for high-priority risks
- Assign risk owners
- Review risk status in OODA loops
- Treat pre-mortem output as requirements

---

## OODA Anti-Patterns

### 19. Skipping Orient
**Pattern:** Moving directly from Observe to Decide without interpretation.

**Symptoms:**
- Data presented without analysis
- "The numbers are X" without "and that means Y"
- Mental models never examined
- Same biases repeated

**Recovery:**
- Explicitly state interpretation
- Challenge assumptions
- Consider alternative explanations
- Ask: "What are we assuming?"

---

### 20. Slow OODA in Fast Context
**Pattern:** Methodical loops when rapid iteration needed.

**Symptoms:**
- Competition moving faster
- Opportunities missed while analyzing
- Perfect plans for outdated situations
- "By the time we decided, it was too late"

**Recovery:**
- Match loop speed to context
- Accept imperfect information
- Smaller, faster experiments
- Decide and adjust, don't wait and plan

---

### 21. Fast OODA in Slow Context
**Pattern:** Rapid iteration when deep analysis is needed.

**Symptoms:**
- Strategic decisions made impulsively
- No time for proper orientation
- Thrashing between approaches
- Activity mistaken for progress

**Recovery:**
- Recognize strategic vs tactical
- Allow time for complicated analysis
- Distinguish "being agile" from "being hasty"
- Some decisions warrant slow OODA

---

### 22. No Loop (O-O-D-A Stop)
**Pattern:** Executing once without reviewing and iterating.

**Symptoms:**
- No review points set
- "We already decided this"
- Ignoring feedback signals
- Sunk cost driving continuation

**Recovery:**
- Set explicit review dates
- Define pivot triggers in advance
- Treat decisions as hypotheses
- Reward adaptation, not stubbornness

## Recovery Patterns Summary

| Anti-Pattern | Quick Recovery |
|--------------|----------------|
| Framework Worship | Match weight to stakes |
| Analysis Paralysis | Set deadline and stick to it |
| Process Theater | Acknowledge when decision is made |
| Complex as Complicated | Design probes, expect emergence |
| Multiple D's | Escalate to find single D |
| Everyone is A | Reserve for true compliance |
| Gaming the Matrix | Define criteria before scoring |
| Superficial Pre-Mortem | Require 10+ failure reasons |
| Skipping Orient | Explicitly state interpretation |
| No Loop | Set review points before acting |

## Self-Check Questions

Before concluding any decision process, ask:

1. Did we classify the domain correctly?
2. Is there exactly ONE D who accepts accountability?
3. Did criteria exist before we scored options?
4. Did we genuinely stress-test, including uncomfortable scenarios?
5. Do we have concrete next steps with owners?
6. When will we review and potentially pivot?

If any answer is unclear, address before proceeding.
